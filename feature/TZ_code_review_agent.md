## ТЗ: Агент автоматического Code Review для MultiAgent

Ссылка на источник концепции: [LLM на службе разработки: как мы научили нейросети проводить код-ревью](https://habr.com/ru/companies/ru_mts/articles/939168/)

### 1. Цели и KPI
- **Цели**:
  - Сократить время между коммитом и получением первичной обратной связи по коду.
  - Снизить нагрузку на коллег по ручному ревью, повысить единообразие замечаний.
  - Стандартизировать новый код под правила проекта.
- **Ключевые метрики** (целевые ориентиры):
  - +20% к количеству Merge Request на разработчика в командах, использующих агента.
  - −8% дефектов на релиз за счёт раннего выявления проблем.
  - ≥60% применяемость рекомендаций (по количеству принятых фиксов/замечаний).

### 2. Область применения и ограничения
- Поддерживаемые артефакты: исходный код (Python, JS/TS, YAML, Markdown), конфигурации CI/CD, инфраструктурные файлы.
- Анализ по умолчанию — изменённые участки (diff), с опцией расширения контекста до всего файла.
- Не выполняет глубокий динамический анализ (fuzzing, e2e); не заменяет обязательное человеческое ревью в критичных модулях.

### 3. Высокоуровневая архитектура
- **Code Review Agent**: оркестратор анализа и агрегации результатов.
- **Diff Collector**: сбор изменений из Git (pre-commit, MR), формирование контекстов.
- **Prompt Builder**: построение структурированных запросов к LLM с учётом стиля проекта.
- **LLM Connector**: обращение к моделям через существующие абстракции проекта.
- **Policy Engine**: применение правил (severity, allow/deny, пороги, исключения).
- **Result Normalizer**: приведение ответов к единому JSON-формату, ранжирование, дедупликация.
- **Feedback Collector**: сбор пользовательского фидбэка, автообучение промптов и политик.

Интеграции:
- IDE/чат: уведомления, предпросмотр замечаний, быстрые фиксы.
- Git/CI: запуск по хукам и в пайплайнах, комментирование в MR.
- Мультиагентная система проекта: профиль агента, запуск из `workflow_pipelines`, использование `custom_tools`.

### 4. Источники данных и контекст
- Git diff, полный текст целевых файлов, ближайшие контексты (соседние функции/классы).
- Конфигурации проекта (линтеры, форматтеры, правила безопасности).
- Исторические замечания и принятые фиксы для адаптации подсказок.

### 5. Триггеры запуска
- `pre-commit` (быстрая локальная проверка на изменённые файлы).
- Создание/обновление MR/PR (полный отчёт и комментарии по строкам при возможности).
- Ручной запуск из чата/CLI/IDE.
- Периодические проверки веток (опционально).

### 6. Функциональные требования
- Анализ изменённых участков с возможностью расширения до всего файла.
- Категории проверок: безопасность, производительность, читаемость/поддерживаемость, соответствие стилю, архитектурная согласованность.
- Ранжирование по важности: Critical, High, Medium, Low, Info.
- Формирование предложений по исправлению (quick-fix/patch) там, где это уместно.
- Умение «объяснить замечание»: краткая аргументация и ссылки на правила.
- Учет настройки проекта (исключения директорий/файлов, глосса проекта, приоритеты правил).
- Отчёт в MR/PR и в локальной сессии разработчика; экспорт в Markdown/JSON.

### 7. Интеграции с экосистемой проекта
- Мультиагентная система:
  - Новый профиль агента: `agent_profiles/code_reviewer.yaml` (описание задач, инструменты, политики).
  - Пайплайн: интеграция в `workflow_pipelines/` (например, шаги «Analyze diffs» и «Comment MR»).
  - Возможность запуска как отдельного шага из `agent_command.py`/чата.
- Git/CI:
  - Скрипты хуков (pre-commit) и задания в CI (GitHub/GitLab) для автозапуска.
  - Публикация результата: комментарии в MR, артефакты пайплайна, статус-check.
- IDE/чат:
  - Уведомление о найденных замечаниях, предпросмотр диффов правок, принятие/отклонение предложений.

### 8. Конфигурация
- Глобально и на репозитории: файл `.codereview.yaml` (severity thresholds, включённые категории, игнор-паттерны, лимиты размеров, модель LLM).
- Пер-проектные гайды код-стиля: подключение внутренних правил и best practices.
- Пороговые значения для чисел замечаний/критичности, влияющих на блокировку MR (опционально).

### 9. Модели и промптинг
- Использовать существующие абстракции проекта для обращения к LLM, ретраев и таймаутов.
- Структурированный промпт: цели, контекст (diff, фрагменты файла, правила), формат ответа (JSON), лимиты токенов.
- Сокращение контекста: семплинг по важности, сжатие нерелевантных частей, чанкинг.
- Вывод без раскрытия цепочек рассуждений; обязательная структурированная часть для машинной обработки.

### 10. Формат результата (JSON Schema, концептуально)
```json
{
  "summary": {"files": 0, "findings": 0, "critical": 0, "high": 0},
  "findings": [
    {
      "id": "CRV-0001",
      "file": "path/to/file.py",
      "line": 123,
      "endLine": 130,
      "category": "security|performance|style|architecture|maintainability",
      "severity": "critical|high|medium|low|info",
      "title": "Краткое название проблемы",
      "message": "Описание проблемы и почему это важно",
      "suggestion": "Короткое предложение по исправлению",
      "patch": "diff/унифицированный формат или псевдокод изменений",
      "confidence": 0.0,
      "autocorrectable": true,
      "references": ["docs/links", "rules/names"]
    }
  ]
}
```

### 11. Производительность и масштабируемость
- Лимиты на размер анализируемых файлов и суммарный контекст; батчирование запросов.
- Кэширование результатов на уровне чанков/файлов для повторяющихся проверок.
- Параллелизм по файлам при локальных проверках; очереди в CI.

### 12. Безопасность и приватность
- Редакция чувствительных данных при формировании промптов (секреты, ключи, персональные данные).
- Логи и артефакты — без исходных секретов; контроль доступа к отчётам.
- Конфигурируемые политики отправки кода во внешние LLM (on/off, allowlist-модули).

### 13. Наблюдаемость и метрики
- Технические: латентность/ошибки запросов к LLM, длина очередей, таймауты.
- Продуктовые: принятые/отклонённые рекомендации, доля автокорректируемых замечаний, динамика дефектов.
- Журналы для трассировки: идентификаторы коммитов, версий правил, хэши контента (без чувствительных данных).

### 14. Приёмочные критерии
- Тестовый репозиторий с эталонными изменениями и ожидаемыми замечаниями.
- Автозапуск по `pre-commit` и по MR, генерация отчёта и корректная разметка серьёзности.
- Не более X% ложноположительных замечаний на baseline-наборе.
- Успешная интеграция с пайплайном проекта и локальным чатом.

### 15. План релиза
- Этап 1: локальный `pre-commit` и отчёт в чате (MVP, без IDE-комментариев).
- Этап 2: публикация в MR/PR с построчными комментариями (если платформа поддерживает).
- Этап 3: быстрые фиксы (apply patch) и сбор фидбэка для адаптации правил.

### 16. Риски и смягчение
- Галлюцинации модели — обязателен человекоцентричный режим: рекомендации необязательны, фокус на подсветке проблемных мест. Подробнее см. статью-референс: [Habr/МТС](https://habr.com/ru/companies/ru_mts/articles/939168/).
- Перегрузка контекста — агрессивная фильтрация/сжатие, пороги размеров.
- Негативное влияние на скорость CI — параллелизм и кэширование.

### 17. Будущее развитие
- Обучение на внутренних решениях и замечаниях команды; адаптивные подсказки.
- Расширение типов анализов (например, межмодульные зависимости, простые taint-проверки).
- Широкая IDE-интеграция и интерактивные автопочинки.

### 18. Артефакты поставки
- Профиль агента `agent_profiles/code_reviewer.yaml` (описание задач/правил/инструментов).
- Шаги пайплайна в `workflow_pipelines/` для запуска анализа.
- Документация по конфигурации `.codereview.yaml` и интеграциям (Git, CI, чат/IDE).
